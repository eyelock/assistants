# Stable Release Procedure

## The Rule: Always Use a Release Branch

**NEVER open `develop → main` directly.** Always cut a `release/vX.Y.Z` branch first.

Why: `main` accumulates commits after every release (auto-generated file updates, hotfix
forward-ports). Opening `develop → main` directly forces conflict resolution inside `develop`,
which is messy and pollutes its history. A release branch absorbs the conflict once, in
isolation, and the back-merge to develop is small and clean.

## Prerequisites

- All changes merged to `develop`
- CI passing on `develop`
- Quality gate passes locally

## Steps

### 1. Cut the Release Branch

```bash
git checkout develop && git pull
git checkout -b release/v{VERSION}
git push -u origin release/v{VERSION}
```

### 2. Prepare the Release

Make any release-specific changes on the branch (CHANGELOG, version bump, etc.):

```bash
git add <files>
git commit -m "chore: prepare release v{VERSION}"
git push
```

### 3. Open the Release PR (release/vX.Y.Z → main)

```bash
gh pr create --base main --head release/v{VERSION} \
  --title "release: v{VERSION}" \
  --body "Promotes release/v{VERSION} to main for stable release."
```

If GitHub reports conflicts, resolve them on the release branch — **never on develop**:

```bash
git fetch origin
git merge origin/main   # resolve conflicts here, then push
git push
```

Wait for CI to pass, then merge using a **true merge** (not squash, not rebase):

```bash
gh pr checks --watch
gh pr merge --merge
git checkout main && git pull
```

### 4. Pre-Release Checks

```bash
# Run your project's quality gate
make check   # or equivalent
```

All gates must pass before tagging.

### 5. Create Release Tag

```bash
git tag -a "v{VERSION}" -m "Release v{VERSION}"
git push origin v{VERSION}
```

**Never tag without the quality gate passing first.**

### 6. Monitor Automated Release

```bash
gh run list --workflow=release.yml --limit 1
gh run watch <run-id>
gh release view v{VERSION}
# Should show: correct title, assets attached, NOT pre-release
```

### 7. Back-Merge the Release Branch to Develop — MANDATORY

```bash
git checkout -b chore/back-merge-v{VERSION} develop
git merge origin/release/v{VERSION}
git push -u origin chore/back-merge-v{VERSION}
gh pr create --base develop \
  --title "chore: back-merge release/v{VERSION} into develop" \
  --body "Brings release-branch changes and conflict resolutions back to develop."
```

Merge once CI passes, then delete the release branch:

```bash
git push origin --delete release/v{VERSION}
```

### 8. Forward-Port Auto-Generated Files to Develop — MANDATORY

If your CI writes files directly to `main` after a release (e.g., an appcast or feed update
workflow), those are never automatically forward-ported. After verifying the release:

```bash
git checkout -b chore/sync-release-files-v{VERSION} develop
git checkout origin/main -- <auto-generated-file-paths>
git commit -m "chore: sync auto-generated release files for v{VERSION} back to develop"
git push -u origin chore/sync-release-files-v{VERSION}
gh pr create --base develop \
  --title "chore: sync release files for v{VERSION} back to develop"
```

> **Tip:** Steps 7 and 8 can be combined if the CI auto-update has already completed
> before you open the back-merge PR.

## Troubleshooting

**Release workflow fails on CI check:** Fix the issue on `main`, wait for CI, delete the
failed tag, re-tag.

```bash
gh release delete v{VERSION} --yes
git tag -d v{VERSION}
git push origin :refs/tags/v{VERSION}
# Then re-tag once the fix is in place
```

## What NOT to Do

- NEVER open `develop → main` directly — always use a release branch
- NEVER create releases manually with `gh release create`
- NEVER tag without running the quality gate first
- NEVER tag from branches other than `main`
- NEVER skip CI verification
- NEVER skip the back-merge (step 7) or auto-generated file sync (step 8)
